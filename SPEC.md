## 1. BRD — 비즈니스 요구사항

### 1.1 배경 및 목적

대학교 내 과팅 문화는 기존에 개인 SNS(오픈카톡, 에브리타임)를 통해 비공식적으로 이루어졌음.  
비공식 채널 사용으로 인해 신원 불명 사용자, 낮은 신뢰도, 성사율 저조 문제 발생.  
**indeed**는 학생증 인증 기반의 신뢰도 높은 과팅 매칭 플랫폼을 제공함.

### 1.2 비즈니스 목표

| 목표 | 지표 | 목표값 |
|------|------|--------|
| 안전한 매칭 환경 제공 | 학생증 인증률 | 100% (인증 없이 이용 불가) |
| 활성 사용자 확보 | MAU | 론칭 3개월 내 500명 |
| 매칭 성사율 제고 | 신청 후 매칭 성사율 | 70% 이상 |
| 플랫폼 신뢰도 유지 | 신고 처리율 | 48시간 내 100% |

### 1.3 타겟 사용자

- **1차 타겟**: 충북대학교 재학생 (남/여, 1~4학년)
- **2차 타겟**: 추후 타 대학교 확장 가능한 구조
- **연령**: 19~27세
- **특성**: 스마트폰 친숙, 과팅 경험 있거나 관심 있는 학생

### 1.4 사용자 페르소나

**페르소나 A — 팀장 박지훈 (22세, 컴퓨터공학과 3학년)**
- 과팅을 주도적으로 기획하는 역할
- 같은 과 친구 2명과 팀 구성 후 여자팀과 매칭 원함
- 채팅 중 약속 장소/시간 조율이 불편하면 이탈

**페르소나 B — 팀원 이수연 (21세, 경영학과 2학년)**
- 팀장이 초대한 코드로 합류하는 수동적 역할
- 신원이 확인된 상대방과만 매칭되기를 원함
- 불쾌한 상대 메시지 신고 기능을 중요하게 여김

**페르소나 C — 관리자 (운영자)**
- 학생증 심사, 신고 처리, 패널티 부여
- 대시보드에서 한 번에 여러 심사 처리 가능해야 함

### 1.5 비즈니스 제약사항

- 충북대학교 학생증만 인증 가능 (현재)
- 만 19세 미만 이용 불가
- 학생증 인증 없이는 매칭 기능 전체 차단
- 개인정보 수집 최소화 (이름, 학과, 성별, 학번만)

### 1.6 성공 기준

- 학생증 인증 후 첫 매칭까지 10분 이내
- 매칭 신청 후 24시간 이내 매칭 성사
- 신고 접수 후 48시간 내 처리
- 앱 크래시율 0.1% 미만

## 2. PRD — 제품 요구사항

### 2.1 제품 개요

**앱명**: indeed  
**플랫폼**: Android (iOS 추후 지원 예정)  
**배포 방식**: EAS Build + OTA 업데이트  
**기술 스택**: Expo SDK 54, React Native 0.81.5, Supabase, Hermes JS Engine

### 2.2 핵심 가치 제안

1. **신뢰** — 학생증 인증으로 재학생만 이용 가능
2. **간편함** — 팀 생성 → 신청 → 자동 매칭, 복잡한 설정 없음
3. **안전** — 신고/패널티 시스템으로 불쾌한 경험 최소화
4. **동시 다중 매칭** — 한 명이 최대 3개 과팅 동시 진행 가능

### 2.3 기능 우선순위

| 우선순위 | 기능 | 상태 |
|----------|------|------|
| P0 (필수) | 로그인/회원가입 | ✅ 완료 |

| P0 | 팀 생성/합류 | ✅ 완료 |

## 3. 유저 플로우

### 3.1 신규 유저 온보딩 플로우

```
앱 실행
  └─ 스플래시 화면 (indeed 로고)
       └─ 세션 확인
            ├─ 세션 있음 → 인증 상태 확인
            │    ├─ verified = true → 홈 탭
            │    ├─ verified_status = 'pending' → 심사 대기 화면
            │    └─ verified_status = 'rejected' → 반려 안내 화면
            └─ 세션 없음 → 온보딩 (indeed 소개)
                  └─ 로그인 화면
                        ├─ 기존 회원: 이메일 + 비밀번호 → 홈 탭
                        └─ 신규 회원: 회원가입
                              └─ 프로필 설정 (이름, 학과, 성별, 학번)
                                    └─ 학생증 업로드
                                          └─ 심사 대기 화면 (Realtime 구독)
                                                └─ 승인 시 자동으로 홈 탭 이동
```

### 3.2 팀 생성 플로우 (팀장)

```
홈 탭 (팀 없음 상태)
  └─ 팀 만들기 버튼
        └─ 팀 생성 화면
              ├─ 팀 이름 입력
              ├─ 성별 선택 (남성 / 여성)
              └─ 인원 선택 (2:2 / 3:3)
                    └─ 팀 생성 완료
                          └─ 홈 탭 (팀 있음 상태)
                                └─ 초대 코드 공유 (팀원 모집)
                                      └─ 팀원 합류 대기
```

### 3.3 팀 합류 플로우 (팀원)

```
홈 탭 (팀 없음 상태)
  └─ 코드 입력 버튼
        └─ 초대 코드 입력 화면
              └─ 코드 검증
                    ├─ 유효하지 않음 → 오류 메시지
                    ├─ 팀 꽉 참 → 오류 메시지
                    ├─ 성별 불일치 → 오류 메시지
                    ├─ 학과 불일치 → 오류 메시지 (같은 과만 가능)
                    └─ 합류 성공 → 홈 탭 (팀원으로 표시)
```

### 3.4 매칭 신청 플로우

```
홈 탭 또는 매칭 탭
  └─ 과팅 신청하기 버튼 (팀장만, 팀 인원 충족 시 활성화)
        └─ 사전 검증
              ├─ 활성 채팅방 3개 이상 → 불가 안내
              └─ 팀원 중 활성 매칭 3개 이상인 사람 있음 → 해당 팀원 이름 표시 후 불가
        └─ apply_and_match RPC 호출
              ├─ 즉시 매칭 성사 (상대 팀 대기 중이었음)
              │    └─ 매칭 완료 화면 → 채팅 탭으로 이동
              └─ 대기 상태 (상대 팀 없음)
                    └─ 매칭 대기 화면 (취소 버튼 있음)
                          └─ Realtime 구독으로 매칭 감지 시 자동 이동
```

### 3.5 채팅 플로우

```
채팅 탭
  └─ 채팅방 목록 (진행 중 / 완료됨 탭)
        └─ 채팅방 선택
              └─ 채팅방 화면
                    ├─ 실시간 메시지 송수신
                    ├─ 읽음 처리 (입장 시 자동)
                    ├─ 안 읽은 메시지 수 표시
                    ├─ 발신자 이름 + 학과 표시
                    ├─ 내 팀 / 상대 팀 말풍선 색상 구분
                    │
                    ├─ [뮤트] 토글 → 해당 채팅 알림 끔
                    │
                    ├─ [신고] 버튼
                    │    └─ 신고 대상 선택 (상대 팀 멤버 목록)
                    │         └─ 신고 이유 입력
                    │              └─ 신고 접수 완료
                    │
                    └─ [과팅 완료] 버튼
                          └─ 완료 확인 모달
                                └─ N/M명 신청 현황 표시
                                      └─ 전원 신청 시 채팅 종료
                                            └─ 🎉 완료 알림 모달 → 뒤로 가기
```

### 3.6 관리자 플로우

```
마이 탭 → 관리자 메뉴 (is_admin = true 일 때만 표시)
  └─ 관리자 화면
        ├─ [심사 대기] 섹션
        │    └─ 학생증 카드 목록
        │         └─ 카드 선택 → 상세 화면
        │               ├─ 학생증 이미지 확인
        │               ├─ 승인 버튼 → users.verified_status = 'approved'
        │               └─ 반려 버튼 → 반려 사유 입력 → rejected
        │
        ├─ [승인됨] 섹션 (최근 50명)
        │
        ├─ [반려됨] 섹션 (최근 50명)
        │
        └─ [신고 내역] 섹션
              └─ 신고 카드 선택 → 상세 모달
                    ├─ 신고자 정보
                    ├─ 피신고자 정보
                    ├─ 신고 내용
                    ├─ 피신고자의 채팅 메시지 (최근 20개)
                    ├─ 관리자 메모 입력
                    └─ 처리 버튼
                          ├─ 처리만 (패널티 없음)
                          ├─ 경고 (+1 패널티)
                          ├─ 3일 정지
                          └─ 7일 정지
```

## 5. 기능 상세 명세

### 5.1 인증 (Auth)

**회원가입**
- 이메일 + 비밀번호로 Supabase Auth 회원가입
- 가입 완료 후 프로필 설정 화면으로 이동

**로그인**
- 이메일 + 비밀번호
- 로그인 후 인증 상태에 따라 라우팅 분기

**로그아웃**
- 마이페이지에서 커스텀 모달로 확인 후 로그아웃
- Supabase Auth 세션 종료 후 스플래시로 이동

**세션 유지**
- 앱 재시작 시 Supabase 세션 자동 복원
- 세션 만료 시 로그인 화면으로 이동

---

### 5.2 프로필 설정

**입력 항목**
- 이름 (필수)
- 학과 (필수, 같은 학과 팀원만 합류 가능)
- 성별 (남성/여성, 필수)
- 학번 (선택)

**제약**
- 학교는 충북대학교 고정 (단일 학교 서비스)
- 프로필 설정 완료 후 학생증 업로드로 이동

---

### 5.3 학생증 인증

**업로드**
- 갤러리에서 이미지 선택 (expo-image-picker)
- base64로 변환 후 Supabase Storage에 업로드
- `users.student_card_url` 저장
- `verified_status = 'pending'` 설정

**심사 대기**
- pending.tsx에서 Realtime 구독
- `users` 테이블 UPDATE 감지 → 승인 시 홈으로 자동 이동
- 반려 시 rejected.tsx로 이동 (사유 표시)

**재제출**
- 반려 후 학생증 재업로드 가능
- 재제출 시 `verified_status = 'pending'` 초기화

---

### 5.4 팀 관리

**팀 생성 (팀장)**
- 팀 이름, 성별, 인원(2v2/3v3) 설정
- 6자리 랜덤 초대 코드 자동 생성
- 기존 팀이 있으면 자동 삭제 후 재생성 (단, 매칭 중이면 오류)

**팀 합류 (팀원)**
- 초대 코드 입력
- 검증: 팀 존재 여부, 인원 초과, 성별 일치, 학과 일치
- 합류 후 team_members 테이블에 'member' role로 삽입

**초대 코드 공유**
- `indeed://join-team?code=XXXXXX` 딥링크 형식
- 클립보드 복사 또는 공유 기능

**팀원 강퇴 (팀장 전용)**
- 팀원 칩의 X 버튼 → 확인 Alert → team_members 삭제
- 매칭 중일 때는 강퇴 불가

**팀 나가기 (팀원)**
- 팀원: team_members에서 본인 삭제
- 팀장: 다음 팀원에게 팀장 이양 후 나가기, 팀원이 없으면 팀 해체

**팀 해체 (팀장)**
- team_members 전체 삭제 → teams 삭제

---

### 5.5 매칭 시스템

**신청 조건**
- 팀 인원이 설정 인원(2 또는 3명) 충족
- 본인 + 팀원 모두 활성 매칭이 3개 미만
- 활성 채팅방 3개 미만

**매칭 로직 (`apply_and_match` SQL RPC)**
- 같은 성별 제외, 같은 인원 조건의 대기 팀 탐색
- FOR UPDATE 잠금으로 동시성 보장
- 대기 팀 있으면 즉시 매칭 (matches INSERT), 없으면 status = 'applied'

**매칭 해제**
- 대기 중일 때 취소 가능 (status = 'waiting'으로 변경)
- 매칭 후 취소 불가

**동시 매칭 제한**
- 사용자 1인당 활성(active) 매칭 최대 3개
- 팀원 중 1명이라도 초과 시 팀 신청 불가

---

### 5.6 채팅

**메시지 송수신**
- Supabase Realtime postgres_changes 구독
- 메시지 insert 즉시 화면에 반영 (낙관적 업데이트)

**발신자 표시**
- 이름 + 학과 표시
- 내 팀원: 파란 계열 배경
- 상대 팀: 회색 배경
- 나: 검은 배경 (오른쪽 정렬)

**읽음 처리**
- 채팅방 입장 시 `mark_messages_read` RPC 호출
- 상대방 메시지의 `read_by` 배열에 내 UUID 추가
- 읽음 표시: 메시지 옆 숫자 (read_by.length)
- 안 읽은 메시지 수: 채팅 목록, 탭 바 배지에 표시

**뮤트**
- 채팅방 뮤트 토글 → `matches.muted_by` 배열에 추가/제거
- 뮤트 시 해당 채팅 알림 미전송 (Edge Function에서 처리 예정)

**채팅 만료**
- `matches.expires_at` 기준으로 채팅 종료 예정 시간 표시
- 만료 시 자동으로 완료됨 처리

**과팅 완료**
- 완료 버튼 클릭 → 하단 시트 모달 (N/M명 신청 현황)
- `toggle_dating_completion` RPC로 원자적 처리
- 전원 신청 시 `matches.status = 'completed'`
- 🎉 완료 알림 모달 → 채팅 목록으로 이동

---

### 5.7 약속 장소/시간 (부분 구현)

**현재 상태**: API(`updateMeetingInfo`, `getMatchDetail`)는 구현되어 있으나 채팅 UI에서 제거된 상태

**필요한 구현**
- 채팅방 내 약속 탭 또는 상단 배너
- 장소, 날짜, 시간 입력 → `matches.meeting_place`, `meeting_date`, `meeting_time` 저장
- 두 팀 모두 확인 가능

---

### 5.8 신고 시스템

**신고 흐름**
1. 채팅방 상단 [신고] 버튼 클릭
2. 상대 팀 멤버 목록에서 신고 대상 선택 (`getMatchMembers`)
3. 신고 이유 텍스트 입력
4. `reports` 테이블에 INSERT (`reporter_id`, `target_id`, `match_id`, `content`)

**관리자 처리**
- 신고 상세 모달: 신고자/피신고자 정보 + 피신고자 메시지 최근 20개
- 처리 옵션: 처리만 / 경고 / 3일 정지 / 7일 정지
- 처리 시 `reports.status = 'resolved'` + 선택한 패널티 적용

**패널티**
- 경고: `users.penalties` +1
- 3일/7일 정지: `users.suspended_until` 날짜 설정
- 정지 중 로그인 가능하나 기능 제한 (미구현 — 클라이언트 측 차단 로직 필요)

---

### 5.9 알림

**푸시 알림 (Expo Notifications)**
- 앱 첫 실행 시 권한 요청
- 토큰을 `push_subscriptions` 테이블에 저장
- Supabase Edge Function `notify`가 DB 웹훅 트리거로 발송

**푸시 발송 조건**
| 이벤트 | 발송 대상 |
|--------|-----------|
| 매칭 성사 | 두 팀 전원 |
| 새 메시지 | 해당 채팅 참여자 중 발신자 제외 |
| 학생증 승인/반려 | 해당 유저 |

**인앱 알림**
- Edge Function이 `notifications` 테이블에도 동시 저장
- 알림 내역 화면에서 확인 가능
- 헤더 벨 아이콘에 안 읽은 수 배지

**푸시 알림 온/오프**
- 마이페이지 토글로 제어
- 끔: `push_subscriptions`에서 삭제
- 켬: `registerPushToken()` 재호출

## 6. 데이터 모델

### 6.1 주요 테이블

```
users
├─ id (uuid, PK)
├─ email
├─ name
├─ university
├─ department
├─ gender
├─ student_id
├─ grade
├─ verified (bool)
├─ verified_status (pending | approved | rejected)
├─ student_card_url
├─ card_submitted_at
├─ rejection_reason
├─ is_admin (bool)
├─ penalties (int, 기본값 0)
├─ suspended_until (timestamptz)
└─ expo_push_token (레거시, push_subscriptions로 이전)

teams
├─ id (uuid, PK)
├─ leader_id → users.id
├─ team_name
├─ department
├─ gender (남성 | 여성)
├─ size (2 | 3)
├─ status (waiting | applied | matched)
├─ invite_code (6자리 대문자+숫자)
└─ created_at

team_members
├─ id (PK)
├─ team_id → teams.id
├─ user_id → users.id
└─ role (leader | member)

matches
├─ id (uuid, PK)
├─ team_a → teams.id
├─ team_b → teams.id
├─ status (active | completed)
├─ muted_by (text[], 뮤트한 유저 UUID 목록)
├─ meeting_confirmed_by (text[])
├─ dating_completed_by (text[])
├─ meeting_place
├─ meeting_date
├─ meeting_time
├─ expires_at
└─ created_at

messages
├─ id (uuid, PK)
├─ match_id → matches.id
├─ sender_id → users.id
├─ content
├─ read_by (text[], 읽은 유저 UUID 목록)
└─ created_at

notifications
├─ id (PK)
├─ user_id → users.id
├─ type (match | chat | info | verify)
├─ title
├─ content
├─ is_read (bool)
└─ created_at

reports
├─ id (PK)
├─ reporter_id → users.id
├─ target_id → users.id
├─ match_id → matches.id
├─ content
├─ status (pending | resolved)
├─ admin_note
└─ created_at

push_subscriptions
├─ id (PK)
├─ user_id → users.id (unique)
└─ expo_push_token
```

### 6.2 핵심 SQL 함수

| 함수명 | 역할 |
|--------|------|
| `apply_and_match(my_team_id)` | 매칭 신청 + 즉시 매칭 시도 (원자적) |
| `mark_messages_read(match_id, user_id)` | 읽음 처리 (sender_id::text 캐스팅 필요) |

| `toggle_dating_completion(match_id, user_id)` | 과팅 완료 토글 |

## 7. 비기능 요구사항

### 7.1 성능

- 채팅 메시지 지연: 1초 이내 (Realtime)
- 매칭 신청 응답: 3초 이내
- 이미지 업로드: 5초 이내 (학생증)
- 앱 콜드 스타트: 3초 이내

### 7.2 동시성

- `apply_and_match`는 PostgreSQL `FOR UPDATE` 잠금으로 레이스 컨디션 방지
- 동일 팀에 대한 동시 신청 → 한 건만 처리, 나머지 대기
- 최대 예상 동시 사용자: 200명 (충북대 초기 서비스 기준)

### 7.3 보안

- Supabase Row Level Security (RLS) 적용
- 학생증 이미지: 서명된 URL (3600초 만료)로만 접근
- anon key는 클라이언트에 노출 (Supabase 표준), service role key는 Edge Function에서만 사용
- 관리자 기능: `users.is_admin = true` 서버 측 검증

### 7.4 안정성

- OTA 업데이트: 네이티브 변경 없으면 APK 재빌드 없이 즉시 배포
- OTA 채널: 항상 `preview` 브랜치 (빌드 채널과 일치)
- 오류 처리: 모든 API 호출 try-catch, 사용자에게 친화적 오류 메시지

### 7.5 플랫폼

- 최소 Android 버전: Android 10 (API 29)
- JS 엔진: Hermes (`crypto.getRandomValues` 없음 → `Math.random()` 사용)
- iOS: 미지원 (향후 추가 예정)
